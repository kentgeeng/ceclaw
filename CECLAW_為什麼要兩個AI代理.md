# CECLAW：為什麼要兩個 AI 代理？

**版本：v1.1 | 日期：2026-04-09**

---

## 一句話結論

> **一個 AI 管不了兩件性質完全相反的事。**
>
> OpenClaw 是公司的法務與人事部門，嚴謹標準對所有員工一致。
> Hermes 是每個員工的私人秘書，記得你的習慣和進度。
> shared_bridge 就是秘書幫你去問人事部門——效率最高，且不會讓秘書弄亂人事部門的檔案。

## 核心差異對照

```
                OpenClaw                    Hermes
                ────────                    ──────
角色            公司法務 + 人事部門          員工私人秘書
狀態            Stateless（每次重置）        Stateful（跨對話記憶）
記憶範圍        全員共用（公司知識）          只屬於本人（個人偏好/進度）
使用者          IT / Admin                  每一位員工
知識來源        RAG 三層 + memory-wiki       vault + session memory
執行環境        OpenShell sandbox（高風險）   日常工具（file/web/terminal）
適合任務        系統管理/部署/法規查詢        個人工作流程/任務執行
```

## 五個維度的完整論據

### 1. 權限污染（最根本）

物理上分開兩層，**權限邊界在 API 層級就定死了**。

這比在 prompt 裡要求模型「請不要洩漏個資」安全一萬倍。

單一 AI 同時持有所有員工的個人記憶，在處理某員工查詢時，其他員工的私人資料可能被帶入 context 導致洩漏。分層之後，員工 A 的薪資協商記錄物理上就不在員工 B 的 Hermes 實例裡，無從洩漏。

```
單一 AI 的風險：
  員工A私人記憶 ──→ 同一個 AI ──→ 員工B查詢時被帶入 context 導致洩漏 ✗

雙層架構：
  員工A vault ──→ Hermes A（只有A）  ─┐
  員工B vault ──→ Hermes B（只有B）  ─┤→ 共用 OpenClaw（公司知識）
  公司知識    ──→ OpenClaw（全員）   ─┘
  隱私邊界在 API 層定死，無法洩漏 ✓
```

### 2. 狀態混亂（Context Contamination）

OpenClaw stateless，每次都是**乾淨的執行環境**。

合在一起的話，模型執行 IT 部署指令時，context 裡可能還殘留著員工剛才聊的私事，關鍵任務出錯率上升。

```
合在一起的 context 污染：
  [今天晚餐吃什麼] [員工A的工作習慣] [重啟所有容器指令] [昨天的任務進度]
  → AI 注意力分散，執行關鍵指令時出錯率上升 ✗

雙層分工後：
  OpenClaw context：[重啟所有容器指令]  → 純淨，執行標準化 ✓
  Hermes  context：[員工習慣][任務進度] → 個人化，連續性強 ✓
```

### 3. 關注點分離（資產與執行環境隔離）

OpenClaw 做最危險的事（exec / git / sandbox），**搞掛了重啟就好**，不影響核心資料。

Hermes 守護 vault 和 memory，這是企業最值錢的資產，**不應暴露在高風險執行環境裡**。

```
高風險執行（OpenClaw）     企業資產（Hermes）
──────────────────────     ─────────────────
exec / terminal            vault/working-context.md
git clone / push           vault/project-state.md
OpenShell sandbox          vault/decisions-log.md
搞掛 → 重啟就好 ✓          損毀 → 無法復原 ✗

→ 必須隔離，不能讓高風險執行環境碰到資產層
```

### 4. 水平擴展潛力

```
現在（POC）：
  Hermes ──→ Router proxy ──→ RAG 注入 ──→ Gemma 4

未來（B70 多租戶）：
  Hermes A ──→ OpenClaw（公司A）
  Hermes B ──→ OpenClaw（公司B）
  Hermes C ──┘

更遠的未來（一對多調度）：
  一個 Hermes（大腦）
    ├──→ OpenClaw Node 1（Python 環境）
    ├──→ OpenClaw Node 2（Node.js 環境）
    └──→ OpenClaw Node 3（另一台伺服器）
  Hermes 不需要關心底層環境如何切換 ✓
```

shared_bridge 是標準協議，不是硬編碼，部署位置可以各自最佳化。

### 5. 體驗統一（後端分層，前端無感）

```
員工視角（感覺只有一個 AI）：
  員工 ──→ Hermes UI
           「公司病假幾天？」
           Router proxy 自動注入 RAG 結果 → Gemma 4 → 回答
           員工完全不知道背後的分層邏輯 ✓

IT 視角（完整控制權）：
  IT ──→ OpenClaw Admin :3005
         管理知識庫 / 更新政策 / 監控系統狀態
```

## 四個場景：為什麼不能只用一套

| 場景 | 只用 Hermes | 只用 OpenClaw | 雙層分工（正確設計）|
|------|-------------|----------------|---------------------|
| 員工問：「我上週進度到哪？」 | ✅ vault 記憶 | ❌ 每次重置，不記得 | Hermes 從 vault 回覆 |
| IT：「重啟所有開發容器」 | ❌ 權限不足，且汙染員工記憶 | ✅ stateless 標準執行 | OpenClaw 執行 |
| 員工問：「公司病假規定？」 | ❌ 個人記憶裡沒有公司政策 | ✅ RAG 知識庫回答 | Router proxy 注入 RAG，Gemma 4 回答 |
| 員工：「整理昨天的對話筆記」 | ✅ 跨對話記憶 | ❌ stateless，無歷史 | Hermes 做 |
| IT 部署新政策給所有員工 | ❌ 每個 Hermes 實例各自為政 | ✅ memory-wiki 全員同步 | OpenClaw 更新，全員 Hermes 自動獲取 |

## 銷售對話劇本

**Q1：「為什麼要兩個 AI？直接用一個 Gemma 4 API 不就好了？」**

> 「因為一個 AI 管不了兩件性質完全相反的事。
>
> 第一，**權限邊界**。公司知識全員共用，員工記憶只有本人。合在一起，員工的私人資料會被帶入 context，資安直接出問題。
>
> 第二，**狀態管理**。IT 系統任務需要每次重置確保標準化；員工助手需要跨對話記憶不重複交代。一個 AI 沒辦法同時做到這兩件事。
>
> 第三，**體驗統一**。後端分層，但透過 Router proxy 串起來。員工感覺只面對一個 AI，實際上背後是兩套獨立邏輯在運作。這才是企業級安全架構。」

**Q2：「那我用 ChatGPT Teams 不就好了？」**

> 「你的資料會出你的內網。CECLAW 全部在你的伺服器，法規、員工記錄、公司知識永不出門。台灣製造業和醫療是硬需求，雲端方案直接出局。」

**Q3：「員工要切換兩個視窗嗎？」**

> 「不需要。員工只看到 Hermes 一個介面。需要查法規，Router 在背後自動注入 RAG 結果，員工完全無感知。」

## 最終定位

```
這不是在增加複雜度，而是在建立安全邊界。

OpenClaw = 公司的「官方知識庫 + 執行引擎」
           嚴謹、標準、對所有員工一致

Hermes   = 每個員工的「私人秘書」
           了解你的習慣、記得你的進度

Router proxy = 自動注入公司知識給每一次對話
           員工無感知，知識隨時在線

這是「分層治理」架構。
生產環境追求的不是全能，而是可控與安全。
```
